# Review

이 문서는 Nudge-Signal Backend 에서 코드 리뷰를 요청하거나 수행할 때 따르는 기준이다. 리뷰는 칭찬이나 요약보다 **버그, 회귀 위험, 보안, 테스트 누락**을 먼저 찾는 절차다.

## 기본 원칙

- 리뷰 요청도 작업이므로 시작 전 반드시 Plan Mode 로 진입한다.
- 리뷰 대상 범위를 먼저 확인한다: 현재 브랜치 전체, 특정 PR, 특정 파일, 특정 커밋 중 무엇인지 명확히 한다.
- 모든 GitHub Pull Request 리뷰 코멘트는 한국어로 작성한다.
- 변경 작성자 의도를 추측하지 않는다. diff, 테스트, 실행 경로, 기존 컨벤션을 근거로 판단한다.
- findings 우선으로 응답한다. 중대한 문제가 있으면 요약보다 먼저 파일/라인과 함께 제시한다.
- 문제가 없으면 "발견한 문제 없음"을 명확히 말하고, 남은 테스트 공백이나 잔여 리스크만 짧게 남긴다.
- 취향, 스타일, 대규모 리팩토링 제안은 버그 위험과 분리한다. 요청 범위를 벗어난 개선은 강요하지 않는다.

## 리뷰 전 확인

1. `git status --short --branch` 로 현재 브랜치와 미커밋 변경을 확인한다.
2. PR 리뷰라면 `gh pr view` / `gh pr diff` 로 base, head, 변경 파일, CI 상태를 확인한다.
3. 로컬 변경 리뷰라면 `git diff` 와 필요 시 `git diff --cached` 를 확인한다.
4. 프로젝트 규칙은 `AGENTS.md`, `.codex/Rule.md`, `.codex/Skill.md` 를 기준으로 삼는다.
5. 신규 코드나 리팩토링이면 `.codex/agents/architecture-alignment.toml` 의 레이어 기준도 함께 본다.

## 우선순위

리뷰 findings 는 아래 순서로 찾고 정렬한다.

1. **P0**: 배포 즉시 장애, 데이터 손상, 인증/권한 우회, 시크릿 노출.
2. **P1**: 주요 기능 회귀, 트랜잭션/동시성/영속성 오류, 운영 환경에서 높은 확률로 터지는 문제.
3. **P2**: 특정 입력이나 경계 조건에서 깨지는 버그, 테스트 누락으로 회귀 가능성이 큰 문제.
4. **P3**: 유지보수성, 명명, 작은 일관성 문제. 단, 실제 위험이 있을 때만 finding 으로 올린다.

## 중점 체크리스트

- **API / Controller**
  - DTO 를 반환하고 엔티티를 직접 노출하지 않는가?
  - 입력 검증(`@Valid`, Bean Validation)이 빠지지 않았는가?
  - HTTP 상태 코드와 에러 응답이 의도와 맞는가?

- **Service / Transaction**
  - 쓰기 작업에 트랜잭션 경계가 있는가?
  - 조회 전용 메서드는 `@Transactional(readOnly = true)` 인가?
  - 비즈니스 규칙이 컨트롤러가 아닌 서비스/도메인에 있는가?

- **Domain / JPA**
  - 엔티티에 `@Data`, 무분별한 `@Setter` 가 없는가?
  - lazy loading 이 컨트롤러 밖으로 새지 않는가?
  - equals/hashCode, 연관관계, 생성 메서드가 JPA 특성과 충돌하지 않는가?

- **Infra / Config**
  - 외부 의존(DB, Redis, 외부 API)이 `infra` 경계 뒤에 있는가?
  - 환경변수 fallback, placeholder, 시크릿 값이 저장소에 들어오지 않았는가?
  - prod/dev/local 설정 차이가 의도와 맞는가?

- **Tests**
  - 변경된 동작을 검증하는 단위/슬라이스 테스트가 있는가?
  - `@WebMvcTest` 등 슬라이스 테스트에 필요한 프로퍼티가 주입됐는가?
  - 실패/경계/권한 케이스가 빠지지 않았는가?

## 출력 형식

리뷰 결과는 아래 순서를 따른다.

1. Findings: 심각도 높은 순서. 각 항목은 파일/라인, 문제, 영향, 재현 조건 또는 근거를 포함한다.
2. Open Questions: 판단에 필요한 질문이 있을 때만.
3. Test Gaps / Residual Risk: 실행하지 못한 검증이나 남은 리스크.
4. Summary: 짧게. findings 가 있으면 부차 정보로만 둔다.

## Inline Finding

Codex 앱에서 라인 코멘트를 남길 때는 `::code-comment{...}` 형식을 사용한다. 한 finding 에 하나의 코멘트만 남기고, 라인 범위는 가능한 좁게 잡는다.

필수 정보:
- `title`: 짧은 제목. 예: `[P1] 트랜잭션 없이 다중 저장 수행`
- `body`: 왜 문제인지, 어떤 상황에서 터지는지, 무엇을 고쳐야 하는지 한 문단으로 작성
- `file`: 절대 경로 또는 워크스페이스 기준 경로
- `start`: 1-based 시작 라인
- `priority`: P0=0, P1=1, P2=2, P3=3
- `confidence`: 0.0-1.0 사이 확신도

## 리뷰 반영 답글

- 리뷰 코멘트의 지적 사항을 코드나 문서에 반영했으면, 해당 코멘트 답글에 정확히 `Done` 이라고 남긴다.
- 일부만 반영했거나 다른 방식으로 해결했다면 `Done` 만 남기지 않는다. 무엇을 반영했고 무엇이 남았는지 짧게 설명한다.
- 반영하지 않기로 결정했다면 근거를 답글에 남기고, 필요한 경우 후속 이슈나 별도 PR 로 분리한다.

## 리뷰가 아닌 것

- 단순 요약만 하는 응답
- 근거 없는 "좋아 보입니다" 승인
- 요청 범위를 벗어난 대규모 리팩토링 강요
- CI 통과만으로 충분하다고 보는 판단
